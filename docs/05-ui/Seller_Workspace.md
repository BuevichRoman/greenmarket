# GreenMarket Seller Workspace

**Version:** 1.0
**Status:** Normative

**Основание:** PRD-000, [Catalog_Template.md](../02-domain/Catalog_Template.md) (PR-008), [Seller_Profile.md](../02-domain/Seller_Profile.md) (PR-009).

Настоящий документ (ТЗ-010) не описывает отдельный программный модуль — он объединяет результаты PR-008 (Google Sheets Template v1.0) и PR-009 (Seller Profile) в единую нормативную спецификацию пользовательского рабочего пространства продавца Stage 1: рабочей книги Google Sheets (Google Sheets spreadsheet), реализующей весь пользовательский интерфейс продавца средствами Google Sheets. После утверждения документа рабочая книга может быть реализована без дополнительных архитектурных решений.

## 1. Назначение

Определить полный пользовательский интерфейс продавца Stage 1, реализованный средствами Google Sheets: структуру рабочей книги, назначение каждого листа, правила заполнения, UX, ограничения и взаимодействие с Publication Pipeline.

## 2. Область ответственности

Документ определяет:

- структуру рабочей книги;
- назначение каждого листа;
- связи между листами;
- правила заполнения;
- требования к UX;
- требования к защите данных;
- требования к совместимости с Publication Pipeline.

Документ не определяет внутреннюю архитектуру Parser, Validator, Mapper, Publication Service — они определяются [Publication_Service.md](../04-services/Publication_Service.md), [Publication_Workflow.md](../04-services/Publication_Workflow.md), [REST_API.md](../04-services/REST_API.md).

## 3. Архитектура Workspace

Workspace — единая рабочая книга Google Sheets. Каждый лист отвечает за отдельную предметную область. Рабочая книга является пользовательским интерфейсом продавца, а не внутренней моделью GreenMarket: она не хранит служебных данных публикации, кроме версии шаблона (лист `_System`, раздел 9).

## 4. Структура рабочей книги

| Лист | Назначение | Владелец данных | Источник данных | Пользователь | Редактируется продавцом | Участвует в публикации |
|---|---|---|---|---|---|---|
| Инструкция | Правила работы с рабочей книгой | GreenMarket | Статический контент шаблона | Продавец (чтение) | Нет | Нет |
| Профиль продавца | Отображение данных Seller Profile | GreenMarket ([Seller_Profile.md](../02-domain/Seller_Profile.md)) | Регистрация продавца / администратор | Продавец (чтение) | Нет | Нет |
| Каталог | Ассортимент продавца | Продавец (пользовательские поля) / GreenMarket (`SellerProductId`) | Продавец заполняет, `SellerProductId` присваивает сервер | Продавец | Да (кроме `SellerProductId`) | Да |
| Товарные группы | Справочник `ProductGroup` | GreenMarket | Domain Model / база данных | Продавец (чтение, источник для выпадающих списков) | Нет | Нет |
| Товарные позиции | Справочник `Product` | GreenMarket | Domain Model / база данных | Продавец (чтение, источник для выпадающих списков) | Нет | Нет |
| `_System` | Метаданные шаблона | GreenMarket | Генерируется сервером при формировании рабочей книги | Скрыт от продавца | Нет | Только проверка версии (раздел 9), не источник данных публикации |

«Товарные группы» и «Товарные позиции» — два отдельных листа (так реализовано в нормативном артефакте `catalog_template_v1.xlsx`, PR-008); «Справочники» в разделе 8 — их общее собирательное имя, а не третий отдельный лист.

## 5. Лист «Инструкция»

**Структура и содержание:** краткие разделы «Как добавить товар», «Как изменить товар», «Как удалить товар», «Как выбрать товарную группу/позицию», «Как опубликовать каталог», «Что делать при ошибке публикации» — без дублирования деталей API или внутренней логики сервисов, только пользовательские шаги. Ссылки на конкретные экраны/ошибки — на [Seller_MVP.md](Seller_MVP.md) и [Publication_Workflow.md](../04-services/Publication_Workflow.md) не даются (продавцу они не нужны), только текстовое описание.

**Примеры:** отсутствуют в виде заполненных строк на этом листе (во избежание попадания тестовых данных в реальный каталог продавца при невнимательном редактировании); примеры заполнения — только в `templates/examples/` для разработчиков (см. [Catalog_Template.md](../02-domain/Catalog_Template.md)), продавцу не передаются.

**Правила обновления:** содержимое листа обновляется вместе с `TemplateVersion`; отдельного версионирования не имеет.

## 6. Лист «Профиль продавца»

Полностью соответствует [Seller_Profile.md](../02-domain/Seller_Profile.md).

**Расположение полей:** одна строка — одно поле, две колонки («Поле», «Значение»), состав и порядок — по разделам 4–5 Seller_Profile.md.

**Обязательные поля:** визуально выделены поля, обязательные для Stage 1 (Seller_Profile.md, раздел 5).

**Комментарии:** краткая подсказка (Note) у полей, формат которых не очевиден без объяснения (например, `workingHours`).

**Проверки ввода, правила отображения ошибок:** не применимо — лист доступен только для чтения (см. ниже).

**Редактирование.** Лист не редактируется продавцом и полностью защищён. Причина: редактирование профиля продавца прямо исключено из объёма Seller_Profile.md (раздел 11: «редактирование профиля», «личный кабинет продавца» — не входит в PR-009); первичный источник данных профиля — регистрация продавца и правки администратора (Seller_Profile.md, раздел 6), а не рабочая книга каталога. Значения на листе GreenMarket проставляет сам при формировании/обновлении рабочей книги (раздел 12).

Если в будущем потребуется самостоятельное редактирование профиля продавцом через Workspace — это отдельное решение вне объёма данного документа.

## 7. Лист «Каталог»

Полностью соответствует [Catalog_Template.md](../02-domain/Catalog_Template.md): структура колонок, обязательность, проверки ввода (Excel/Sheets Data Validation с `showErrorMessage=True`), подсказки — по разделу «Структура рабочего каталога» и колонкам, зафиксированным в `structure_validator.py` (`CATALOG_COLUMNS`). Выпадающие списки колонок «Товарная группа GreenMarket» / «Товарная позиция GreenMarket» ссылаются на листы «Товарные группы» / «Товарные позиции» (раздел 8).

## 8. Лист «Справочники»

Собирательное имя для листов «Товарные группы» (`ProductGroup`) и «Товарные позиции» (`Product`).

**Источник формирования:** сервер заполняет оба листа актуальными данными при каждом формировании новой редакции рабочей книги (см. Catalog_Template.md, «Обновление рабочего каталога»).

**Защита:** оба листа полностью защищены от редактирования продавцом.

**Использование в Data Validation:** единственное назначение — источник списков для выпадающих списков листа «Каталог» (раздел 7). Фактическая проверка принадлежности выбранной группы/позиции при публикации выполняется сервером по данным базы данных, а не по содержимому этих листов (см. раздел 11) — содержимое листов может отставать от базы данных на время между формированиями редакций, поэтому не используется как источник истины при валидации.

## 9. Лист `_System`

Определяет только:

- `TemplateVersion`;
- `TemplateId`.

Другие данные публикации (`PublicationKey`, `CatalogHash`, внутренние идентификаторы) на листе не хранятся — см. раздел 13 и [Catalog_Template.md](../02-domain/Catalog_Template.md) (CR-001). Лист скрыт и защищён от редактирования.

## 10. Пользовательский UX

Единые правила оформления для всей рабочей книги:

- **Оформление и цвета:** обязательные поля выделены визуально (например, полужирным заголовком колонки); поля, доступные только для чтения (Профиль продавца, Справочники, `_System`), — единым нейтральным фоном, отличающим их от редактируемых.
- **Комментарии и подсказки:** комментарии (Note) на заголовках колонок/полей, где формат или смысл не очевиден без объяснения.
- **Выпадающие списки:** Data Validation с `showErrorMessage=True` (см. Catalog_Template.md — критический баг, найденный ревью PR-008, где часть проверок не показывала ошибку) везде, где значение должно выбираться из справочника.
- **Защита диапазонов:** защищены все нередактируемые листы/диапазоны (Профиль продавца, Справочники, `_System`, `SellerProductId` в «Каталоге»).
- **Закрепление строк:** заголовок листа «Каталог» закреплён (Freeze Row) для сохранения видимости при прокрутке.
- **Фильтрация:** на листе «Каталог» включён базовый автофильтр по колонкам.
- **Ширина колонок:** подобрана по содержимому, без обрезки текста по умолчанию.

Цель — сделать заполнение максимально понятным для продавца без обучения.

## 11. Совместимость с Publication Pipeline

| Лист | Читается Parser | Проверяется Validator | Участвует в Mapper | Влияет на публикацию |
|---|---|---|---|---|
| Инструкция | Да | Только присутствие листа | Нет | Нет |
| Профиль продавца | Нет | Нет | Нет | Нет |
| Каталог | Да | Да (структура и бизнес-правила) | Да | Да |
| Товарные группы | Да | Да (только структура: обязательные колонки) | Нет | Нет — используется только для UX (раздел 8); проверка существования группы выполняется по базе данных |
| Товарные позиции | Да | Да (только структура) | Нет | Нет — аналогично |
| `_System` | Да | Да (обязательные поля, поддерживаемая версия шаблона) | Нет | Да — `TemplateVersion` проверяется до остальных этапов пайплайна |

Лист «Профиль продавца» не читается `GoogleSheetsParser` для целей публикации каталога — публикация каталога и Seller Profile имеют независимые жизненные циклы (Seller_Profile.md, раздел 2). Добавление листа «Профиль продавца» в нормативный артефакт `catalog_template_v1.xlsx` и генератор `backend/app/catalog_template/build.py` — реализационная задача вне объёма данного документа (см. раздел 12).

## 12. Версионирование

Изменение структуры рабочей книги (состав листов, состав или порядок колонок, состав служебных полей) сопровождается:

- изменением `TemplateVersion`;
- обновлением настоящего документа и связанных нормативных документов (Catalog_Template.md, Seller_Profile.md — при изменении их разделов, на которые ссылается Workspace);
- обновлением тестов Parser/Validator/Mapper;
- проверкой совместимости `StructureValidator` (`SUPPORTED_TEMPLATE_VERSIONS`) с новой версией.

Реализация листа «Профиль продавца» в нормативном артефакте (раздел 11) требует отдельного изменения `TemplateVersion` и не выполняется в рамках настоящего документа.

## 13. Ограничения

Workspace не содержит:

- `PublicationKey`;
- `CatalogHash`;
- внутренних идентификаторов публикаций;
- вычисляемых системой данных, кроме `TemplateVersion`/`TemplateId` (раздел 9).

Все вычисления выполняются GreenMarket.

## 14. Не входит в документ

- Apps Script;
- макросы;
- автоматическая публикация;
- синхронизация;
- WebHook;
- личный кабинет продавца.

## 15. Критерии приёмки

Работа считается завершённой, если:

- создана рабочая книга Google Sheets (Seller Workspace v1.0);
- структура рабочей книги соответствует настоящему документу;
- реализованы все предусмотренные листы;
- лист «Профиль продавца» соответствует [Seller_Profile.md](../02-domain/Seller_Profile.md);
- лист «Каталог» соответствует [Catalog_Template.md](../02-domain/Catalog_Template.md);
- лист `_System` содержит только `TemplateVersion` и `TemplateId`;
- реализованы все проверки ввода, выпадающие списки, комментарии, защита диапазонов и другие элементы UX, предусмотренные настоящим документом;
- рабочая книга корректно импортируется из нормативного `catalog_template_v1.xlsx` (пока он остаётся эталоном) либо является официальным шаблоном Google Sheets (если принято считать его эталоном);
- Publication Pipeline успешно принимает данные, подготовленные в рабочей книге, без дополнительных изменений;
- продавец может самостоятельно создать копию рабочей книги, заполнить её и подготовить каталог к публикации.

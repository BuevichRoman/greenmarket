# GreenMarket Customer UI UX — Покупатель (Stage 1)

**Version:** 1.0
**Status:** Draft (черновик, ещё не отправлен на ревью коллеге)

**Основание:** [Buyer_MVP.md](Buyer_MVP.md) (структурный контракт: экраны, элементы, ограничения Stage 1), [Domain_Model.md](../02-domain/Domain_Model.md) (Product, ProductGroup, SellerProduct), [REST_API.md](../04-services/REST_API.md) (раздел Catalog API). Решение по объёму (17.07.2026): сценарий Stage 1 строится строго на уже согласованной предметной модели и не вводит новых доменных сущностей (Basket, Household, Purchase Option, Shopping Route, карта, Bottom Sheet) — это предмет отдельного архитектурного этапа (Stage 2), начинающегося с расширения Domain Model, а не с UX.

**Разграничение с [Buyer_MVP.md](Buyer_MVP.md):** [Buyer_MVP.md](Buyer_MVP.md) — структурный контракт (состав экранов, элементы, правила видимости предложений, ограничения); настоящий документ — сценарий использования этих экранов покупателем (что видит первым, как перемещается между экранами, как ищет и сравнивает). Требования к составу экранов и правилам отображения здесь не переопределяются, только описывается поведение вокруг них.

Настоящий документ — черновик, подготовленный без готовой спеки/макета от коллеги (по его прямому указанию: «спеки нет, накидываем вариант сами»), по аналогии со структурой [Seller_Workspace_UX.md](Seller_Workspace_UX.md). В отличие от Seller Workspace UX, который сверялся с уже задеплоенным и провалидированным артефактом, здесь сверяться не с чем — Catalog API, которым должен пользоваться Buyer Web, определён нормативно в [REST_API.md](../04-services/REST_API.md), но не реализован в `backend/` (см. раздел 10, пункт 2). Документ описывает ожидаемое поведение против контракта, а не проверенный на коде UX.

## 1. Назначение

Определить пользовательский сценарий покупателя в Buyer Web: что покупатель видит на каждом из трёх экранов Stage 1, как перемещается между ними, как ищет и сравнивает предложения, и на каком шаге сценарий Stage 1 заканчивается.

## 2. Область ответственности

Документ определяет:

- сценарий работы покупателя от открытия Buyer Web до выбора продавца;
- UX-детали каждого из трёх экранов Stage 1;
- навигацию между экранами;
- поведение поиска и категорий;
- состояния экрана, прямо определённые [Buyer_MVP.md](Buyer_MVP.md) (отсутствие фото, непубликуемые остатки), и перечень состояний, которые им не определены.

Документ не определяет состав и структуру экранов (см. [Buyer_MVP.md](Buyer_MVP.md)), визуальный дизайн (макеты Figma/v0.dev — отдельная UI Design Specification, явно выведена за рамки [Buyer_MVP.md](Buyer_MVP.md)) и внутреннюю логику Catalog Service/REST API.

## 3. Сценарий работы покупателя

1. Покупатель открывает Buyer Web — без регистрации, весь каталог доступен анонимно (Stage 1 не требует авторизации). Точка входа всегда одна — экран «Главная», сохранённого состояния между сессиями нет (в Stage 1 отсутствуют избранное, история, персонализация).
2. Покупатель ищет товар (строка поиска) или переходит по дереву категорий / популярным категориям — попадает на экран «Каталог товаров».
3. На экране «Каталог товаров» покупатель видит список товаров, может уточнить поиск, сменить категорию или отсортировать список, не возвращаясь на «Главная».
4. Покупатель открывает интересующий товар — попадает на экран «Карточка товара», видит список предложений продавцов по этому товару.
5. Покупатель сравнивает предложения (цена, единица измерения, остаток, фото, описание) и выбирает подходящего продавца.
6. **Сценарий Stage 1 считается завершённым после того, как покупатель выбрал продавца для интересующего товара.** Дальнейшее взаимодействие с продавцом (контакт, оформление заказа, доставка, оплата) находится вне области ответственности Customer UI Stage 1 — не отсутствует по недосмотру, а сознательно выведено за границу документа (формулировка согласована с коллегой 17.07.2026). [Buyer_MVP.md](Buyer_MVP.md) явно исключает корзину и оформление заказа из MVP.

## 4. UX экрана «Главная»

Элементы экрана согласно [Buyer_MVP.md](Buyer_MVP.md): строка поиска, дерево категорий, популярные категории, последние опубликованные товары (опционально).

Экран — единственная точка входа покупателя, не персонализирован (нет авторизации, нет истории показов). Взаимный визуальный приоритет элементов (что показано крупнее — поиск или категории) нормативно не определён — вопрос отдельной UI Design Specification, вне области этого документа.

## 5. UX экрана «Каталог товаров»

Список товаров, доступный двумя путями: поиск или выбор категории на «Главная», либо уточнение поиска/категории прямо на этом экране (Buyer_MVP.md явно допускает оба действия здесь же, без возврата на «Главная»).

Каждый товар в списке отображает: фотографию, название, минимальную цену среди предложений, количество продавцов. В расчёт минимальной цены и количества продавцов попадают только предложения, прошедшие правила видимости — продавец активирован и не деактивирован, `SellerProduct` опубликован, `Product` активен (Buyer_MVP.md, §«Предложения продавцов»). Товар без ни одного видимого предложения в списке не появляется.

Поиск группирует результаты по `Product` — если у товара несколько продавцов, покупатель сначала видит один товар, а не список продавцов напрямую.

Сортировка — минимум по названию, по цене, по популярности при наличии данных (Buyer_MVP.md, §«Сортировка»).

## 6. UX экрана «Карточка товара»

Список предложений продавцов по одному товару. Для каждого предложения: продавец, цена, единица измерения, остаток (если продавец его публикует — см. раздел 9), фотографии (`SellerProduct`, при отсутствии — заглушка платформы), описание товара продавца.

Основная задача экрана — сравнение: Definition of Done Stage 1 формулирует цель именно как «сравнить предложения нескольких продавцов; выбрать подходящего продавца» (Buyer_MVP.md). На выборе продавца сценарий Stage 1 завершается (см. раздел 3, шаг 6) — дальнейшее взаимодействие с продавцом сознательно вне области ответственности этого документа.

## 7. Навигация между экранами

- Линейный переход вперёд: Главная → Каталог товаров → Карточка товара.
- «Каталог товаров» достижим и с «Главная» (поиск/категория), и напрямую с самого себя — сменой поиска или категории без возврата.
- «Назад» — стандартная навигация браузера (Buyer Web — мобильный веб, не отдельное приложение), отдельного механизма назад/breadcrumbs не описано.
- Прямые ссылки на конкретный товар или категорию (для шаринга) — не определены, открытый вопрос (см. раздел 10, пункт 4).

## 8. UX поиска и категорий

- Поиск выполняется по `Product`, не по продавцу или предложению — результаты группируются по товару (раздел 5).
- Категории — навигация по `ProductGroup`, единая структура каталога GreenMarket, не зависящая от конкретного продавца.
- Момент выполнения поиска (на каждый символ / по вводу с debounce / по нажатию) нормативно не определён — вопрос реализации и UI Design Specification.

## 9. Состояния экрана

Прямо определено [Buyer_MVP.md](Buyer_MVP.md):

- нет фото у предложения → отображается заглушка платформы;
- продавец не публикует остатки → соответствующая информация не отображается вовсе (не «0», не прочерк — просто отсутствует).

Не определено, открытые вопросы (раздел 10, пункт 5):

- пустая выдача (нет результатов поиска, нет товаров в категории);
- загрузка данных, офлайн, ошибка сети;
- гонка — все предложения товара становятся недоступны (продавцы деактивированы) уже после того, как покупатель открыл карточку товара.

## 10. Открытые вопросы

1. ~~Что происходит после выбора продавца.~~ **Закрыто 17.07.2026:** сценарий Stage 1 формально завершается на выборе продавца, дальнейшее взаимодействие с продавцом (контакт, заказ, доставка, оплата) — вне области ответственности Customer UI Stage 1 (см. раздел 3, шаг 6), а не просто неопределено.
2. **Catalog API не реализован.** `GET /api/v1/catalog/groups`, `GET /api/v1/catalog/products`, `GET /api/v1/catalog/products/{id}` определены нормативно в [REST_API.md](../04-services/REST_API.md), но в `backend/app/api/v1/` есть только Publication API (`publications.py`) — Catalog API отсутствует в коде на 17.07.2026. Этот документ описывает ожидаемое поведение против контракта, не проверено на реальном коде.
3. **Фото товара в списке при нескольких продавцах.** [Buyer_MVP.md](Buyer_MVP.md) и [REST_API.md](../04-services/REST_API.md) говорят про «фотографии» в ответе `GET /api/v1/catalog/products`, но не определяют правило выбора, чьё фото показывается на плитке товара, если у разных продавцов разные фотографии одного `Product`.
4. **Deep link на товар/категорию** — не определён, полезен для шаринга, но вне текущей нормативки.
5. **Debounce поиска, состояния загрузки/пустой выдачи/офлайн** — не определены Buyer_MVP.md, вопрос реализации и отдельной UI Design Specification.

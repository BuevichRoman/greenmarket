# GreenMarket Seller Workspace UX

**Version:** 1.0
**Status:** Normative

**Основание:** [Seller_Workspace.md](Seller_Workspace.md) (ТЗ-010), [Catalog_Template.md](../02-domain/Catalog_Template.md) (PR-008), `backend/app/catalog_template/build.py` (единственный источник фактической реализации UX книги).

**Разграничение с [Seller_Workspace.md](Seller_Workspace.md):** [Seller_Workspace.md](Seller_Workspace.md) — контракт рабочей книги (структура листов, поля, требования); настоящий документ — сценарии использования этой рабочей книги продавцом (открытие, заполнение, навигация, публикация). Требования к структуре и полям здесь не переопределяются, только описывается поведение вокруг них.

Настоящий документ детализирует пользовательский сценарий работы продавца с рабочей книгой Seller Workspace v1.0, структура которой определена [Seller_Workspace.md](Seller_Workspace.md) (ТЗ-010): как продавец открывает книгу, что видит в первую очередь, как заполняет каждый лист, как перемещается между листами и как понимает, что каталог готов к публикации. Описание соответствует уже задеплоенному и провалидированному артефакту (`catalog_template_v1.xlsx`, PR-008, реальная рабочая книга Google Sheets, проверена 14.07.2026), а не только нормативному тексту — там, где реализация расходится с нормативной спекой, это явно отмечено в разделе 10.

Лист «Профиль продавца» в документе не описывается — он не входит в состав Seller Workspace v1.0 (см. [Seller_Workspace.md](Seller_Workspace.md), раздел 6).

## 1. Назначение

Определить пользовательский опыт (UX) продавца при работе с рабочей книгой Google Sheets: сценарий открытия и заполнения, UX каждого листа, навигацию внутри книги, поведение при заполнении (подсказки, обязательные поля, выпадающие списки, защита, ошибки) и сценарий публикации.

## 2. Область ответственности

Документ определяет:

- сценарий работы продавца с книгой от открытия до публикации;
- UX-детали каждого видимого продавцу листа;
- навигацию и визуальные ориентиры внутри книги;
- поведение элементов заполнения (подсказки, валидация ввода, защита);
- то, как продавец узнаёт результат публикации.

Документ не определяет состав листов и их структуру (см. [Seller_Workspace.md](Seller_Workspace.md)), формат колонок листа «Каталог» (см. [Catalog_Template.md](../02-domain/Catalog_Template.md)) и внутреннюю логику Parser/Validator/Mapper/Publication Service.

## 3. Сценарий работы продавца

1. Продавец получает ссылку на официальный шаблон GreenMarket и создаёт собственную копию («Файл» → «Создать копию») — работает только в своей копии.
2. Продавец расшаривает свою копию на Service Account GreenMarket (e-mail указан в личном кабинете продавца) с правом «Читатель» — без этого шага публикация невозможна, GreenMarket не сможет прочитать книгу.
3. При открытии книги активным по умолчанию является **лист «Каталог»** — он идёт первым в порядке вкладок слева направо: Каталог → Товарные группы → Товарные позиции → Инструкция (`_System` скрыт и в списке вкладок не отображается). Продавец видит таблицу для заполнения сразу, а не инструкцию — если это нежелательно, порядок вкладок можно изменить в `build.py` (см. раздел 10).
4. Продавец заполняет лист «Каталог» (раздел 5), при необходимости обращается к листу «Инструкция» (раздел 4) или к справочникам «Товарные группы»/«Товарные позиции» как к источнику значений для выпадающих списков.
5. Когда книга заполнена, продавец переходит в личный кабинет продавца (Seller Cabinet, см. [Seller_MVP.md](Seller_MVP.md)), вставляет ссылку на свою таблицу и запускает публикацию (раздел 9).
6. После публикации сервер формирует новую редакцию книги (см. [Catalog_Template.md](../02-domain/Catalog_Template.md), «Обновление рабочего каталога») — продавец продолжает работу с новой редакцией, не с исходной копией.

## 4. UX листа «Инструкция»

Лист защищён от редактирования (`ws.protection.sheet = True`), состоит из одной широкой колонки (ширина 100) со статическим текстом, разбитым на пять пронумерованных разделов: как сделать копию шаблона, как заполнить каталог, как расшарить таблицу на Service Account, как опубликовать каталог, типичные ошибки. Продавец только читает — интерактивных элементов на листе нет.

Содержание листа зафиксировано в `backend/app/catalog_template/data.py` (`INSTRUCTION_LINES`) и обновляется вместе с `TemplateVersion` (см. [Seller_Workspace.md](Seller_Workspace.md), раздел 5).

## 5. UX листа «Каталог»

Одна строка = один товар. Заголовок каждой колонки цветом сигнализирует её статус:

| Колонка | Статус | Цвет заголовка | Источник значения |
|---|---|---|---|
| `SellerProductId` | служебное, скрыто от продавца (колонка A скрыта) | серый | присваивается сервером |
| Наименование продавца | обязательное | жёлтый | вводится вручную |
| Товарная группа GreenMarket | обязательное | жёлтый | выпадающий список |
| Товарная позиция GreenMarket | необязательное | зелёный | выпадающий список или «Прочее»; можно оставить пустым |
| Цена | обязательное | жёлтый | вводится вручную, проверка ≥ 0 |
| Единица продажи | обязательное | жёлтый | выпадающий список (можно ввести своё значение — см. ниже) |
| Остаток | обязательное | жёлтый | вводится вручную, проверка ≥ 0 |
| Описание | необязательное | зелёный | вводится вручную |
| Дополнительные характеристики | необязательное | зелёный | вводится вручную |

Каждый заголовок колонки несёт комментарий (Note) с подсказкой и примером значения (`COLUMN_HINTS` в `data.py`) — продавец видит его при наведении на маленький индикатор в углу ячейки заголовка, без необходимости открывать отдельную инструкцию.

Добавление нового товара — просто новая строка без заполненного `SellerProductId`. Изменение — правка полей существующей строки (сопоставление по `SellerProductId`, не по названию). Удаление строки эквивалентно снятию товара с публикации.

## 6. UX листа `_System`

Лист скрыт от продавца (`sheet_state = "hidden"`) и защищён от редактирования. Продавец с ним не взаимодействует — содержит только `TemplateVersion` и `TemplateId`, которые сервер проверяет при публикации (см. [Catalog_Template.md](../02-domain/Catalog_Template.md)).

## 7. Навигация внутри книги

- **Порядок вкладок:** Каталог → Товарные группы → Товарные позиции → Инструкция (`_System` не отображается в списке вкладок).
- **Цвет как сигнал состояния:** жёлтый — обязательно к заполнению, зелёный — необязательно, серый — только для чтения/служебное (см. раздел 5). Единая цветовая схема используется только на листе «Каталог» — заголовки справочников и `_System` целиком серые (лист целиком только для чтения).
- **Переходы между листами** выполняются продавцом вручную через вкладки Google Sheets — книга не содержит внутренних гиперссылок или кнопок навигации между листами.
- **Связь со справочниками:** значения выпадающих списков колонок «Товарная группа GreenMarket» / «Товарная позиция GreenMarket» на листе «Каталог» совпадают с содержимым листов «Товарные группы» / «Товарные позиции» — технически это два независимых представления одних и тех же данных (`data.py`), а не список, ссылающийся на диапазон ячеек этих листов (см. раздел 10). Продавцу разница не видна: оба листа при каждой генерации книги формируются из одного источника и не расходятся.

## 8. UX заполнения

- **Подсказки:** комментарии на заголовках колонок листа «Каталог» (раздел 5).
- **Обязательные поля:** визуально выделены жёлтым фоном заголовка; сервер отдельно проверяет их непустоту при публикации (`SemanticValidator`) — визуальная подсказка не заменяет серверную проверку.
- **Выпадающие списки:** реализованы для «Товарная группа GreenMarket», «Товарная позиция GreenMarket» и «Единица продажи» через Data Validation (Excel/Sheets), с собственным заголовком и текстом ошибки для каждого:
  - «Товарная группа GreenMarket» и «Товарная позиция GreenMarket» — `showErrorMessage=True`, ввод значения не из списка блокируется («Неизвестная товарная группа» / «Неизвестная товарная позиция»);
  - «Единица продажи» — список из типичных единиц (кг, г, шт, уп, л, мл), но с `errorStyle="warning"`: продавец видит предупреждение («Необычная единица продажи»), но может ввести собственное значение и продолжить — единственная нестрогая валидация на листе.
  - Цена/Остаток — числовая валидация (Data Validation, decimal ≥ 0) с блокирующим сообщением об ошибке, но не выпадающий список.
- **Защита:** справочники и «Инструкция» защищены целиком (`protection.sheet = True`) — продавец физически не может редактировать чужие данные. Колонка `SellerProductId` на листе «Каталог» только скрыта, отдельной защиты ячеек в текущей реализации нет (см. раздел 10).
- **Сообщения об ошибках ввода** (внутри самой таблицы, до публикации) — see выше, показываются Excel/Sheets в момент ввода значения, не запрашивают сервер.

## 9. UX публикации

Публикация выполняется не внутри книги (в книге нет ни Apps Script, ни кнопки публикации — см. [Seller_Workspace.md](Seller_Workspace.md), раздел 14), а через личный кабинет продавца (Seller Cabinet):

1. Продавец считает книгу готовой, когда заполнил все обязательные поля по всем добавленным/изменённым товарам — сама книга не показывает продавцу никакого индикатора готовности (ни статуса, ни счётчика ошибок) до отправки на публикацию.
2. В Seller Cabinet продавец вставляет ссылку на свою таблицу (`sheet_url`) и запускает публикацию — `POST /api/v1/publications` (см. [REST_API.md](../04-services/REST_API.md)). Запрос синхронный: продавец получает результат в рамках одного ответа, без ожидания и промежуточных статусов.
3. **Успех:** ответ содержит `success: true`, `publication_id`, количество созданных/обновлённых/снятых с публикации товаров (`created`/`updated`/`deactivated`) и сообщение «Публикация выполнена успешно».
4. **Ошибка валидации (HTTP 422):** ответ содержит `error.code = "VALIDATION_ERROR"`, `error.message = "Каталог не прошёл валидацию"` и `error.details` — список объектов `{sheet, row, column, message}`, по одному на каждую невалидную строку/поле каталога (лист, номер строки, колонка и текст ошибки).
5. После исправления ошибок в книге продавец повторяет публикацию той же ссылкой — `PublicationKey` не меняется, если ошибка не связана с версией шаблона (см. [Catalog_Template.md](../02-domain/Catalog_Template.md), «Правила работы продавца»).

## 10. Известные расхождения с нормативной спекой

Обнаружено при подготовке этого документа (сверка [Seller_Workspace.md](Seller_Workspace.md) и REST API с фактическим кодом), не исправлено в рамках этой задачи:

- ~~§10 [Seller_Workspace.md](Seller_Workspace.md) заявляет, но `build.py` не реализует: закрепление заголовка, автофильтр, подбор ширины колонок~~ — **исправлено**: `freeze_panes = "A2"`, `auto_filter` на весь диапазон данных (`A1:I1000`), явная ширина каждой колонки листа «Каталог» (`COLUMN_WIDTHS`, `backend/app/catalog_template/data.py`) — TDD, `catalog_template_v1.xlsx` перегенерирован и обновлён в репозитории (не в уже задеплоенной production Google-таблице — её пересоздание/переимпорт не выполнялось, это отдельное ручное действие).
- **§10 заявляет защиту диапазона `SellerProductId`,** но в коде колонка A только скрыта (`hidden=True`), явной защиты ячеек нет — это осознанный trade-off, а не недосмотр: лист «Каталог» намеренно не защищён целиком (`test_catalog_sheet_is_not_sheet_protected`), чтобы продавец мог свободно добавлять/удалять строки; частичная защита одной колонки при незащищённом листе в Excel/Sheets ненадёжна и не стоит риска сломать возможность редактирования. При снятии скрытия колонка технически редактируема — сервер всё равно не доверяет `SellerProductId` из файла без проверки (см. [Catalog_Template.md](../02-domain/Catalog_Template.md), «Защита служебных данных»).
- ~~Ошибки публикации теряют структуру между `ValidationError` и API-ответом~~ — **исправлено**: `error.details` в ответе `POST /api/v1/publications` теперь возвращает объекты `{sheet, row, column, message}` (TDD-фикс, `backend/app/api/v1/schemas.py`/`publications.py`, тест `test_validation_errors_include_sheet_row_column`), а не голые строки. Соответствует [Seller_MVP.md](Seller_MVP.md) («Экран 4. Ошибки») в части строки/колонки; поля «код ошибки» и «рекомендация по исправлению» как отдельные структурные поля по-прежнему не выделены — есть только `message`.
- **Выпадающие списки не ссылаются на листы «Товарные группы»/«Товарные позиции» как на диапазон** — см. раздел 7. Не является дефектом (значения гарантированно совпадают, так как оба списка строятся из одного источника при каждой генерации книги), но противоречит буквальному прочтению [Seller_Workspace.md](Seller_Workspace.md), раздел 8 («источник списков» в смысле живой ссылки на диапазон).

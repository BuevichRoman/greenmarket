# GreenMarket Publication Workflow

**Version:** 1.0
**Status:** Normative

## Назначение

Настоящий документ определяет порядок выполнения публикации рабочего каталога — последовательность этапов от приёма запроса до фиксации результата. Публикация выполняется синхронно, в рамках одного HTTP-запроса и одной транзакции базы данных (см. [Publication_Service.md](Publication_Service.md), раздел "Выбор синхронной обработки"), и не имеет собственного долгоживущего жизненного цикла.

Документ описывает только процесс публикации каталога. Жизненный цикл Product, SellerProduct и Seller определяется отдельными документами. Состояние клиентского документа (рабочего каталога Excel) и механизм `PublicationKey` описаны в [Catalog_Template.md](../02-domain/Catalog_Template.md).

## Назначение workflow

Publication Workflow обеспечивает: однозначный порядок этапов; атомарность бизнес-процесса; единообразную обработку ошибок. Все публикации проходят одинаковую последовательность этапов и завершаются одним HTTP-ответом.

## Этапы публикации

- **Publication Request** — получен файл и `publication_key`.
- **Structure Validation** — проверка структуры: обязательные листы, колонки, версия шаблона.
- **Business Validation** — проверка данных: обязательные значения, типы, цена, остаток, единицы измерения, ссылки на фотографии.
- **Database Transaction** — сравнение редакций, создание/обновление SellerProduct, обновление фотографий, создание CatalogPublication, обновление Seller. Все операции — одной транзакцией.
- **Commit** — публикация успешно завершена; новая версия каталога становится текущей.

## Схема выполнения

Успешная публикация:

```text
Publication Request
 │
 ▼
Structure Validation
 │
 ▼
Business Validation
 │
 ▼
Database Transaction
 │
 ▼
Commit
```

Публикация с ошибкой:

```text
Publication Request
 │
 ▼
Structure Validation / Business Validation
 │
 ▼
Validation Error → HTTP 422 (каталог не изменяется)
```

Техническая ошибка внутри транзакции:

```text
Database Transaction
 │
 ▼
Rollback → HTTP 5xx (каталог не изменяется)
```

## Rollback

Rollback выполняется только для ошибок, возникших на этапе Database Transaction. Причины:

- ошибка SQL;
- нарушение ограничений;
- системная ошибка;
- ошибка хранения файлов;
- внутренняя ошибка Publication Service.

После rollback состояние каталога полностью соответствует состоянию до начала публикации.

## Ошибки

Ошибки делятся на две категории.

**Ошибки пользователя** (отсутствует обязательная колонка, неверный формат цены, некорректная единица измерения) → ошибка на этапе Structure Validation или Business Validation, ответ `HTTP 422` со списком ошибок, каталог не изменяется.

**Технические ошибки** (ошибка БД, ошибка файловой системы, ошибка сети, внутренняя ошибка сервиса) → ошибка на этапе Database Transaction, полный rollback, ответ `HTTP 5xx`.

## Повторная публикация

Каждый вызов публикации — независимая операция, выполняемая синхронно в рамках одного запроса. Продавец может отправить новую редакцию в любой момент после получения ответа на предыдущую попытку. Повторная обработка одного и того же `publication_key` запрещена (см. [Publication_Service.md](Publication_Service.md), раздел "Идемпотентность").

## Инварианты

Publication Workflow гарантирует: результат публикации — либо полный успех, либо полный откат, промежуточных состояний не существует; ошибки пользователя и технические ошибки обрабатываются раздельно; завершённая публикация не изменяется; опубликованный каталог не бывает частично обновлён; rollback полностью восстанавливает предыдущее состояние.

# Design: Product Card Apps Script (цикл 2 из 3 — карточка товара продавца)

**Дата:** 2026-07-22
**Статус:** Approved (Roman), design для внутренней реализации.

## Контекст

Цикл 1 (см. [design](2026-07-21-photo-upload-backend-design.md)/[plan](../plans/2026-07-21-photo-upload-backend.md)) построил backend-контракт: `POST /api/v1/photos` (загрузка фото продавца в S3, создание `Photo`), обязательную колонку «Фото» в шаблоне (список `Photo.id` через `;`, `TemplateVersion 2.0`), полный Publication Pipeline, синхронизирующий `SellerProductPhoto`. Всё это уже задеплоено и покрыто тестами (236/236).

Настоящий документ — цикл 2: Apps Script внутри рабочей книги продавца, дающий карточку товара (форма всех полей строки + управление фото) без ручного редактирования ячеек и без ручных вызовов API. Первый Apps Script в проекте — тулинга (clasp/CI) под это ещё нет.

## Находки при бренсторме (меняют первоначальные предположения)

1. **`access_token` не хранится в `_System`.** Первая идея — по аналогии с `TemplateVersion`/`TemplateId`. Оказалась нереализуема: сервер физически не пишет ничего обратно в живую Google-таблицу продавца — ни сейчас, ни как задокументированный, но не построенный механизм (`Catalog_Template.md` описывает «сервер формирует новую редакцию файла», но это 0% кода). Решение: `access_token` хранится в `PropertiesService.getDocumentProperties()` скрипта — продавец вводит один раз тот же токен, что уже вводит в Seller Cabinet. Держит цикл 2 без нового backend-кода для этой части.
2. **URL фото по `s3_key` нигде не определён.** Нужен и карточке (превью), и позже витрине покупателя (цикл 3). Решение: S3-бакет публичный на чтение, URL — стандартный паттерн `https://{bucket}.s3.{region}.amazonaws.com/{s3_key}`. Настройка бакета — инфраструктурный шаг вне кода (AWS-консоль, отдельно от этой реализации).
3. **Превью уже прикреплённых (не в текущей сессии карточки) фото невозможно без нового backend-эндпоинта.** `POST /api/v1/photos` возвращает только `photo_id`, не `s3_key`; способа обратного поиска `s3_key` по `photo_id` не существовало. Решение: новый `GET /api/v1/photos?ids=...` (см. ниже) — единственное отступление от «цикл 2 — ноль backend-работы», минимальное и по образцу уже существующих эндпоинтов.

## Scope

**В объём:**
- Новый `GET /api/v1/photos?ids=1,2,3&access_token=...` (backend, минимальный) — возвращает URL фото по списку id, только для фото этого продавца.
- Apps Script, привязанный к рабочей книге продавца: меню «GreenMarket» → «Открыть карточку» (на выделенной строке листа «Каталог») / «Добавить товар» (на новой строке).
- Модальное окно (`HtmlService` + `showModalDialog`) с формой всех пользовательских полей строки: наименование продавца, товарная группа (выпадающий список из листа «Товарные группы»), товарная позиция (выпадающий список из «Товарные позиции» + «Прочее»), цена, единица продажи, остаток, описание, доп. характеристики.
- Секция «Фото»: превью уже прикреплённых фото (через новый `GET /api/v1/photos`) и фото, добавленных в текущей сессии карточки; кнопка «Удалить» на каждом (убирает id из списка в памяти карточки); кнопка «Добавить фото» (file picker → `POST /api/v1/photos` → добавление `photo_id` в список).
- «Сохранить» пишет все поля формы, включая итоговый список фото через `;`, обратно в строку листа «Каталог» (для новой строки — добавляет её в конец).
- Хранение `access_token` в `PropertiesService`, запрашивается у продавца при первом использовании карточки в этой книге.
- Файлы скрипта — в репозитории (`apps_script/product_card/`) для ревью; деплой — ручное копирование в Script Editor рабочей книги (шаги документируются).

**Вне объёма (сознательно):**
- Любая клиентская валидация полей формы (цена ≥ 0 и т.п.) — полагаемся на существующую Data Validation листа и ошибки при публикации (`SemanticValidator`).
- Перестановка фото (drag-and-drop) — только добавить/удалить, порядок = порядок добавления в текущей сессии + существующий порядок для старых.
- `DELETE /api/v1/photos/{id}` — при удалении фото в карточке id просто не попадает в список ячейки; `Photo`/S3-объект остаётся сиротой до следующей публикации (тот же принятый в цикле 1 tradeoff).
- `clasp`/CI для Apps Script — деплой полностью ручной в этом цикле.
- Изменения `TemplateVersion`/структуры шаблона — состав колонок не меняется (колонка «Фото» уже добавлена в цикле 1).
- Изменения существующих эндпоинтов (`POST /api/v1/photos`, Catalog API и т.д.) — используются как есть, новый эндпоинт независим.
- Customer UI / показ фото покупателю — цикл 3.

## Backend: `GET /api/v1/photos`

Новый файл `app/api/v1/photos.py` (тот же модуль, что `POST /api/v1/photos`), дополнительный маршрут в том же роутере.

- **Запрос:** `GET /api/v1/photos?ids=1,2,3&access_token=...`.
- **Авторизация:** `resolve_seller_access(access_token)` — тот же механизм, что везде (`403 SELLER_ACCESS_DENIED` при невалидном токене).
- **Логика:** парсинг `ids` (строка через запятую, целые числа — `422` при мусоре); `PhotoGateway` получает новый метод `list_by_ids_and_seller(photo_ids: list[int], seller_id: int) -> list[tuple[int, str]]` (raw SQL, `SELECT id, s3_key FROM Photo WHERE id IN (...) AND seller_id = :seller_id`, тот же стиль возврата — сырые кортежи, — что и остальной `PhotoGateway`) — это первое реальное использование `Photo.seller_id` (миграция 010) как фильтра доступа, не просто трассируемости. id, принадлежащие другому продавцу или не существующие, молча опускаются в ответе (не ошибка — Apps Script просто не покажет превью для них).
- **URL:** строится сервером из `settings.s3_bucket`/`settings.s3_region` — новая небольшая функция `build_photo_url(s3_key) -> str` (например в `app/platform/photo_storage.py` рядом с остальной S3-логикой), паттерн `https://{bucket}.s3.{region}.amazonaws.com/{s3_key}`.
- **Ответ:** `200`, `{"photos": [{"photo_id": int, "url": str}, ...]}`.

## Apps Script: архитектура

Один Container-bound script (привязан к рабочей книге продавца). Компоненты:

- **`Code.gs`** — `onOpen()` (меню «GreenMarket»); `openCardForSelectedRow()`/`openCardForNewRow()` (открывают модалку); серверные функции, вызываемые из HTML через `google.script.run`:
  - `getCardData()` — читает поля строки + список её `photo_id` (парсинг ячейки «Фото» по `;`), справочники групп/позиций (напрямую `SpreadsheetApp`, без бэкенда);
  - `getPhotoUrls(photoIds)` — вызывает `GET /api/v1/photos` через `UrlFetchApp`, возвращает `{photo_id, url}` для превью;
  - `uploadPhoto(base64Data, contentType)` — вызывает `POST /api/v1/photos` через `UrlFetchApp` (multipart), возвращает `photo_id`;
  - `saveRow(rowIndex, formData)` — пишет поля + список `photo_id` через `;` обратно в строку (новую — добавляет в конец листа «Каталог»);
  - `getOrPromptAccessToken()` — читает `PropertiesService.getDocumentProperties()`, если пусто — показывает `ui.prompt(...)`, сохраняет введённое.
- **`Card.html`** — форма + фото-секция + встроенный JS (`google.script.run.withSuccessHandler(...).withFailureHandler(...)`).
- **`appsscript.json`** — манифест: OAuth scopes `spreadsheets.currentonly`, `script.external_request` (для `UrlFetchApp`), `script.container.ui`.

## Данные и поток

1. Продавец выделяет строку → «GreenMarket → Открыть карточку» → `getCardData()` отдаёт поля строки + справочники → модалка открывается заполненной; для строки с уже прикреплёнными `photo_id` карточка вызывает `getPhotoUrls([...])` → превью.
2. «Добавить товар» — та же модалка, но пустая форма и признак «новая строка».
3. «Добавить фото» — браузерный file picker → файл в base64 → `uploadPhoto(...)` → `POST /api/v1/photos` → новый `photo_id` добавляется в список карточки, превью строится сразу из локально выбранного файла (без повторного похода на `GET /api/v1/photos` — файл уже есть в браузере как base64/blob).
4. «Удалить» на превью — убирает `photo_id` из списка в памяти карточки (никакого сетевого вызова).
5. «Сохранить» — `saveRow(rowIndex, formData)` пишет все поля строки, включая `photo_ids.join(";")` в ячейку «Фото». Модалка закрывается.

## Обработка ошибок

Любая ошибка сетевого вызова (загрузка фото, `getPhotoUrls`, невалидный/просроченный `access_token` → `403`) — `SpreadsheetApp.getUi().alert(текст_ошибки)`, карточка остаётся открытой, ранее введённые в форме данные не теряются (состояние живёт в `Card.html`, не пересоздаётся). При `403` от любого вызова — дополнительно предлагается заново ввести `access_token` (перезаписывает `PropertiesService`).

## Тестирование

- Backend: `GET /api/v1/photos` — стандартный TDD-цикл (`pytest`), как весь остальной backend: тест на успешный ответ, на фильтрацию по `seller_id` (чужой/несуществующий id молча опускается), на `403` при невалидном токене, на `400`/`422` при мусоре в `ids`.
- Apps Script: юнит-тестов нет (нет фреймворка в этом проекте для Apps Script) — только ручное тестирование в реальной привязанной таблице по чек-листу в плане реализации (открыть карточку на существующей/новой строке, добавить/удалить фото, сохранить, проверить ячейку «Фото» и лист «Каталог», проверить последующую публикацию).

## Деплой

Файлы Apps Script хранятся в `apps_script/product_card/` в репозитории — источник для ревью и истории изменений, не для автоматического деплоя. Первое разворачивание и любое обновление — ручное копирование содержимого файлов в Script Editor (`Расширения → Apps Script` в рабочей книге), пошаговая инструкция фиксируется в плане реализации/`Seller_Workspace_UX.md`.

## Инфраструктурное предусловие (вне кода)

S3-бакет (`settings.s3_bucket`) должен быть публичным на чтение (bucket policy/ACL на AWS) — иначе построенные URL не откроются ни в карточке, ни впоследствии на витрине. Это шаг вне этой реализации, на стороне Романа (AWS-консоль), не часть плана.

# GreenMarket Database Coding Standard

**Version:** 1.0
**Status:** Normative

## Назначение

Настоящий документ определяет единые правила проектирования схемы базы данных GreenMarket Stage 1. Цель стандарта — обеспечить единообразие структуры базы данных, повысить читаемость SQL и минимизировать вероятность ошибок при дальнейшем развитии системы. Все новые таблицы, поля, индексы и ограничения должны соответствовать настоящему документу.

## Общие принципы

- Простота важнее универсальности.
- Нормализация важнее дублирования.
- Явные связи предпочтительнее неявных.
- База данных является источником истины.
- Каждая таблица имеет единственную ответственность.

## Именование таблиц

Существительные в единственном числе, `PascalCase`.

Правильно: `Product`, `Seller`, `SellerProduct`, `CatalogPublication`.

Неправильно: `Products`, `SellerProducts`, `CatalogPublications`.

## Именование полей

Стиль `snake_case`. Примеры: `product_group_id`, `seller_product_code`, `current_catalog_version`, `created_at`.

## Первичные ключи

Каждая таблица имеет поле `id` типа `BIGINT UNSIGNED AUTO_INCREMENT`. Во всех таблицах используется одинаковый формат первичного ключа.

## Внешние ключи

Название поля формируется по шаблону `<entity>_id`. Примеры: `seller_id`, `product_id`, `photo_id`, `product_group_id`.

## Типы данных

| Назначение | Тип |
|---|---|
| Идентификаторы | `BIGINT UNSIGNED` |
| Строки | `VARCHAR(n)` — размер по назначению; `VARCHAR(255)` «по умолчанию» запрещён |
| Текст | `TEXT` |
| Деньги | `DECIMAL(p,s)` — `FLOAT`/`DOUBLE` запрещены |
| Логика | `BOOLEAN` (в MySQL — `TINYINT(1)`), префикс `is_` |
| Дата/время | `DATETIME`, всегда UTC |
| Контрольные суммы (SHA-256) | `CHAR(64)` |
| PublicationKey (UUID) | `CHAR(36)` — сознательно не `BINARY(16)`: на первом этапе простота хранения и диагностики важнее производительности |

## Логические поля

Булевы признаки начинаются с префикса `is_`. Примеры: `is_active`, `is_published`.

## Даты

`created_at`, `updated_at`. Исторические таблицы могут использовать только `created_at`.

## NULL

NULL допускается только тогда, когда отсутствие значения имеет самостоятельный бизнес-смысл. NULL не используется вместо значения по умолчанию.

## Индексы

Создаются только `PRIMARY KEY`, `FOREIGN KEY`, `UNIQUE`, индексы для подтверждённых сценариев поиска. Создание индексов «на будущее» запрещается. Новые индексы добавляются только после анализа производительности и результатов `EXPLAIN ANALYZE` (кандидаты — см. приложение к [DDL_Specification.md](DDL_Specification.md)).

## UNIQUE

Используется исключительно для обеспечения бизнес-инвариантов, не для оптимизации запросов.

## CHECK CONSTRAINT

Используется только для простых инвариантов, поддерживаемых MySQL (например, `price >= 0`). Сложные проверки реализуются на уровне приложения.

## Удаление данных

Для бизнес-сущностей физическое удаление не используется. Применяются признаки состояния (`is_active`, `is_published`). Исторические записи не удаляются.

## Исторические таблицы

Исторические таблицы являются неизменяемыми. После создания записи запрещаются `UPDATE` и `DELETE`. Добавление новых записей выполняется только владельцем соответствующего бизнес-процесса. В Stage 1 исторической таблицей является `CatalogPublication`.

## Владение данными

Каждая таблица имеет единственного владельца.

| Таблица | Владелец |
|---|---|
| ProductGroup | Admin |
| Product | Admin |
| SellerProduct | Publication Service |
| SellerProductPhoto | Publication Service |
| CatalogPublication | Publication Service |
| Seller | Platform |

Изменение данных компонентом, не являющимся владельцем, запрещается.

## Служебные поля

Служебные поля используются только соответствующим сервисом. В Stage 1 Publication Service является владельцем следующих полей таблицы Seller: `current_catalog_version`, `current_publication_key`, `current_catalog_hash`. Другие компоненты системы не должны изменять эти поля.

## Статусы

`ENUM` не используется — статусы хранятся как строковые значения (`VARCHAR`), с проверкой набора значений через `CHECK`. Для `moderation_status`: `WAIT_PRODUCT`, `IN_PROGRESS`, `RESOLVED`. Расширение перечня — без изменения структуры БД.

## Триггеры и хранимые процедуры

Триггеры — только для технических задач, бизнес-логика в триггерах запрещена. Publication Service не зависит от хранимых процедур — вся бизнес-логика реализуется на прикладном уровне.

## SQL-миграции

Каждая миграция должна: выполнять одну логическую задачу; быть идемпотентной при повторном развёртывании среды; иметь понятное имя; содержать комментарий с назначением. Миграции должны выполняться строго по порядку.

## Изменение схемы

Изменение существующей схемы базы данных допускается только через новую миграцию. Изменение уже опубликованных миграций запрещается.

## Соответствие стандарту

Все изменения схемы базы данных проходят проверку на соответствие настоящему документу. Любое исключение оформляется через Architecture Decision Record ([ADR](../06-development/adr/README.md)).

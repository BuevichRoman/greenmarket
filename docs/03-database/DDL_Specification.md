# GreenMarket SQL DDL Specification

**Version:** 1.0
**Status:** Normative

## Назначение

Настоящий документ определяет правила физической реализации модели данных GreenMarket Stage 1 в MySQL. Документ является нормативной спецификацией для разработки SQL DDL, миграций и проверки схемы базы данных. DDL должен полностью соответствовать настоящему документу.

## Целевая СУБД

MySQL Community Server **8.0.16+** (первая версия с полной поддержкой `CHECK CONSTRAINT`). Более ранние версии не допускаются.

## Общие требования

**Кодировка:** `utf8mb4` / `utf8mb4_0900_ai_ci` для всех таблиц.

**Движок хранения:** `InnoDB` для всех таблиц, без исключений.

**Первичные ключи:** все таблицы используют суррогатный ключ `BIGINT UNSIGNED AUTO_INCREMENT`. Исключения допускаются только при наличии технического обоснования.

**Временные поля:** все изменяемые таблицы содержат `created_at`, `updated_at`. Исторические таблицы могут содержать только `created_at`.

## Внешние ключи

Все связи реализуются внешними ключами. Использование «логических» связей без `FOREIGN KEY` не допускается.

**ON DELETE:** по умолчанию `RESTRICT` или `NO ACTION`. Каскадное удаление применяется только для технических таблиц (например, связь `SellerProductPhoto → SellerProduct`).

**ON UPDATE:** `CASCADE`, если изменение ключа допускается моделью данных; иначе `RESTRICT`.

## NULL

NULL допускается только тогда, когда отсутствие значения имеет самостоятельный бизнес-смысл (например, `SellerProduct.product_id` до модерации, `Seller.current_catalog_version/current_publication_key/current_catalog_hash` до первой публикации). NULL не используется как замена значения по умолчанию.

## Логическое удаление

Физическое удаление бизнес-данных не используется. Применяются признаки состояния: `is_active`, `is_published`.

## Денежные значения

Все цены хранятся в `DECIMAL`. Использование `FLOAT` и `DOUBLE` запрещено.

## Количество товара

Остатки хранятся в числовом поле (`DECIMAL`, поддерживает дробные единицы измерения — кг, литры). Использование строкового хранения запрещено.

## Строковые идентификаторы

Все бизнес-идентификаторы имеют явно ограниченную длину. Использование `TEXT` для идентификаторов запрещено. Контрольные суммы (SHA-256) — `CHAR(64)`; `publication_key` (UUID) — `CHAR(36)`.

## Индексы

Создаются только индексы, необходимые для первичных ключей, внешних ключей, уникальных ограничений, подтверждённых сценариев поиска. Добавление индексов «на будущее» не допускается. Дополнительные индексы создаются только после анализа фактических запросов и результатов `EXPLAIN ANALYZE`.

### Минимальный обязательный набор (Stage 1)

| Таблица | Индекс |
|---|---|
| Seller | `current_publication_key` (UNIQUE) |
| ProductGroup | `parent_id`, `name` |
| Product | `product_group_id`, `name` |
| SellerProduct | `seller_id`, `product_id`, `moderation_status`, `is_published` |
| SellerProductPhoto | `seller_product_id` |
| CatalogPublication | `(seller_id, version)` (UNIQUE), `publication_key` (UNIQUE) |

### Приложение: кандидаты на дополнительные составные индексы

Не создаются на старте — добавляются отдельной миграцией только после подтверждения конкретного медленного запроса результатами `EXPLAIN ANALYZE`.

| Таблица | Индекс | Сценарий |
|---|---|---|
| SellerProduct | `(seller_id, is_published)` | построение опубликованного каталога продавца |
| SellerProduct | `(moderation_status, seller_id)` | очередь модерации (Admin Service) |
| SellerProduct | `(seller_id, product_id)` | поиск и аналитика |
| Product | `(product_group_id, is_active)` | поиск внутри группы |
| Product | `(product_group_id, name)` | поиск товара по группе и названию |
| ProductGroup | `(parent_id, sort_order)` | построение дерева |
| CatalogPublication | `(seller_id, published_at)` | история публикаций продавца |

Индексы, которые сознательно не создаются без подтверждённого сценария: `catalog_hash`, `current_catalog_version`, `current_catalog_hash` — пока отсутствуют запросы, использующие эти поля для поиска.

## Уникальные ограничения

`UNIQUE` используется только для обеспечения бизнес-инвариантов (например, `publication_key` глобально уникален и используется один раз). `UNIQUE` не заменяет проверку бизнес-логики приложения.

## CHECK

`CHECK CONSTRAINT` используется там, где он поддерживается MySQL и позволяет выразить бизнес-инвариант средствами базы данных (например, `price >= 0`, `stock >= 0`, `moderation_status IN ('WAIT_PRODUCT', 'IN_PROGRESS', 'RESOLVED')`). Сложные проверки реализуются на уровне приложения.

## Служебные поля

Поля, используемые исключительно Publication Service, запрещается изменять другим компонентам системы. К ним относятся: `current_catalog_version`, `current_publication_key`, `current_catalog_hash`.

## История публикаций

CatalogPublication является неизменяемой исторической таблицей. После создания записи запрещаются `UPDATE` и `DELETE`. Добавление новых записей выполняется только Publication Service.

## Текущая версия каталога

Текущее состояние каталога хранится исключительно в таблице Seller. История публикаций хранится исключительно в CatalogPublication. Дублирование этих функций запрещается.

## Производительность

На этапе Stage 1 приоритетами являются, в указанном порядке: корректность модели данных → простота сопровождения → читаемость схемы → производительность. Оптимизация схемы выполняется только после получения измеримых данных эксплуатации.

## Соответствие спецификации

Все SQL-миграции обязаны соответствовать настоящему документу. Любое отклонение должно быть оформлено отдельным Architecture Decision Record ([ADR](../06-development/adr/README.md)).

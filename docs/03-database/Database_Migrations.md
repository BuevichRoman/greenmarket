# GreenMarket Database Migrations

**Version:** 1.0
**Status:** Normative

## Назначение

Настоящий документ определяет правила создания, применения и сопровождения миграций базы данных GreenMarket Stage 1. Документ также определяет правила первоначального заполнения базы данных (Seed Data). Все изменения схемы базы данных выполняются исключительно посредством миграций.

> **Разрешение конфликта нумерации.** В черновиках обсуждения существовало два конкурирующих плана: шесть раздельных `CREATE TABLE` (001–006) и их альтернатива — одна объединённая миграция `001_create_catalog_schema.sql` с иным сдвигом номеров seed-миграций. Настоящий документ и файлы в [`../../database/migrations/`](../../database/migrations/) фиксируют **раздельный вариант (001–006)** как финальный и нормативный. История вопроса задокументирована отдельно и в данный комплект документации не входит.

Исполняемые SQL-миграции хранятся не в документации, а в [`docs/database/migrations/`](../../database/migrations/) — настоящий документ на них ссылается, но не дублирует их содержимое.

## Общие принципы

Каждая миграция должна:

- выполнять одну логическую задачу;
- быть атомарной;
- иметь понятное назначение;
- выполняться строго по порядку;
- не зависеть от ручных действий.

После публикации изменение существующей миграции запрещается — исправления выполняются новой миграцией.

## Порядок применения

```text
001 → 002 → 003 → 004 → 005 → 006 → 007 → 008
```

Изменение порядка выполнения не допускается.

## Состав миграций Stage 1

Исполняемые SQL-файлы — в [`docs/database/migrations/`](../../database/migrations/).

| Файл | Назначение |
|---|---|
| `001_create_product_groups.sql` | создание таблицы `ProductGroup` |
| `002_create_products.sql` | создание таблицы `Product` |
| `003_create_seller.sql` | создание таблицы `Seller` (учётная запись продавца, ссылка на `users.id_user` платформенной БД) |
| `004_create_photo.sql` | создание таблицы `Photo` (метаданные фотографий, временное решение Stage 1 — файлы в S3) |
| `005_create_seller_products.sql` | создание таблицы `SellerProduct` |
| `006_create_seller_product_photos.sql` | создание таблицы `SellerProductPhoto` |
| `007_create_catalog_publications.sql` | создание таблицы `CatalogPublication` |
| `008_alter_seller_catalog.sql` | расширение таблицы `Seller` полями `current_catalog_version`, `current_publication_key`, `current_catalog_hash` |

> **Примечание о нумерации.** Исходный план (Database_Migrations.md v1.0) содержал 001–006 и предполагал выполнение на пустой БД, где `Seller`, `User`, `Photo` — таблицы других доменных сервисов платформы, создаваемые отдельно. При первом реальном деплое на платформенную БД `aristotel_taxi` (сервер 104.171.133.95) выяснилось, что таких таблиц нет ни в одной среде. Решение: `Seller` — своя таблица GreenMarket (владелец Platform, ссылается на `aristotel_taxi.users`), `Photo` — своя временная таблица GreenMarket; `User` отдельной таблицей не заводится, соответствующие поля (`moderator_id`, `published_by`) ссылаются на `aristotel_taxi.users(id_user)` напрямую. Миграции 003–006 (исходная нумерация) сдвинуты на 005–008, добавлены новые 003 (`Seller`) и 004 (`Photo`).

## Правила создания миграций

Каждая миграция должна содержать:

- описание назначения;
- создание или изменение одной логической части модели;
- внешние ключи;
- ограничения;
- необходимые индексы.

Добавление тестовых данных в миграции схемы запрещается — для этого используется отдельный этап Seed Data.

## Правила изменения схемы

Изменение существующих объектов выполняется только новой миграцией. Запрещается:

- изменять опубликованные миграции;
- изменять порядок миграций;
- объединять уже опубликованные миграции.

## Seed Data

После выполнения всех миграций база данных заполняется начальными данными. Seed Data предназначены исключительно для разработки, тестирования, демонстрации — не для production-каталога, который наполняется администраторами и продавцами.

### ProductGroup (минимальное дерево категорий)

```text
Фрукты
  Цитрусовые
  Яблоки
  Бананы
Овощи
  Томаты
  Огурцы
  Картофель
Молочные продукты
  Молоко
  Сыр
  Йогурты
```

### Product (минимальный каталог товаров)

```text
Фрукты: Апельсин, Лимон, Банан, Яблоко
Овощи: Помидор, Огурец, Картофель, Лук
Молочные продукты: Молоко, Сыр, Йогурт
Мясо: Говядина, Курица
Рыба: Сардина, Тунец
Напитки: Минеральная вода
```

Каталог не стремится сразу охватить весь ассортимент рынка — расширяется во время эксплуатации через административную модерацию, без изменения структуры БД.

### Seller / SellerProduct (демо-продавцы и предложения)

`003_demo_platform_users.sql` + `004_demo_sellers_and_offers.sql` наполняют
рабочий каталог продавцов демо-данными для Buyer Web (Catalog API) — три
демо-продавца (`Демо: Ферма Ромашково`, `Демо: Хутор Заречный`, `Демо: Сад
Плодовый`) с опубликованными `SellerProduct`, частично на одни и те же
`Product` по разной цене (проверка сравнения предложений и tie-break при
равной цене).

`003_demo_platform_users.sql` вставляет тестовых пользователей платформы
(`users`) и запускается **только там, где `users` — тестовый стаб**
(локальный dev, CI) — см. заголовок файла. На реальном окружении
(aristotel_taxi) эта таблица платформенная и реальная: `004` рассчитан на то,
что три одноимённых пользователя (`name`) там уже существуют — их заводит
платформа (реальные аккаунты с ролью Водитель, выделенные для демо), не этот
репозиторий.

Photo — тестовый набор изображений, только для локальной разработки (не
реализовано отдельным сидером).

## Требования к Seed Data

Seed Data должны:

- разворачиваться автоматически;
- не содержать производственных данных;
- быть воспроизводимыми;
- позволять немедленно начать разработку интерфейсов.

Все идентификаторы — через `AUTO_INCREMENT`, фиксированные значения `id` не используются.

## Развёртывание новой среды

```text
Создание БД → Миграции → Seed Data → Готовая среда разработки
```

После завершения процедуры база данных полностью готова к работе.

## Проверка миграций

Каждая новая миграция должна успешно выполняться на пустой базе данных, совместно со всеми предыдущими миграциями, без ручного вмешательства.

# GreenMarket Architecture Principles

**Version:** 1.0
**Status:** Normative

## Назначение

Настоящий документ — единственный источник определений архитектурных принципов GreenMarket Stage 1. Остальные документы ссылаются на эти принципы по имени, не повторяя определения.

Любое архитектурное решение первого этапа должно соответствовать всем принципам ниже. Исключение оформляется отдельным Architecture Decision Record ([ADR](../06-development/adr/README.md)).

## Database First

Сначала проектируется модель данных. Затем реализуются сервисы и пользовательские интерфейсы.

## State Before Process

Первичным является состояние системы. Бизнес-процессы строятся поверх модели данных.

## Keep MVP Small

Первый этап содержит только функциональность, необходимую для проверки основной гипотезы (см. [Vision.md](../01-overview/Vision.md)).

## Revenue Before Intelligence

Приоритетом является получение первых платящих пользователей. Расширенная автоматизация откладывается до подтверждения бизнес-модели.

## Listen to Reality

Дальнейшее развитие определяется результатами эксплуатации, а не предположениями.

## Extract on Second Use

Абстракция, обобщение или переиспользуемый компонент выделяются только тогда, когда для них появляется второй реальный сценарий использования — не заранее, «на будущее».

## One Source of Truth

Для каждой категории данных существует единственный источник истины (см. таблицу владения в [Development_Guidelines.md](../06-development/Development_Guidelines.md)). Дублирование ответственности между документами и компонентами запрещается.

## One Owner per Data

Каждая таблица и каждый служебный набор полей имеет единственного владельца, только он вправе их изменять (см. таблицу владения в [Physical_Model.md](../03-database/Physical_Model.md) и [Coding_Standard.md](../03-database/Coding_Standard.md)).

## API First

Все пользовательские интерфейсы работают только через REST API. Прямой доступ UI к базе данных запрещён (см. [REST_API.md](../04-services/REST_API.md)).

## Simplicity Before Flexibility

Минимальная схема, минимальный набор индексов и минимальная абстракция предпочтительнее преждевременной универсальности. Оптимизация и обобщение выполняются только после подтверждённой практикой необходимости.

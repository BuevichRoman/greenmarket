# GreenMarket Stage 1 Roadmap

**Version:** 1.0
**Status:** Normative

## Назначение

Настоящий документ определяет порядок реализации первого этапа GreenMarket. Документ фиксирует последовательность создания компонентов системы и критерии завершения каждого этапа разработки.

## Общий принцип

Проект реализуется по принципу **Database First**. Каждый следующий компонент строится поверх уже реализованного предыдущего. Последовательность разработки изменению не подлежит без архитектурного решения (ADR).

## Этап 1. Database

**Цель:** создать физическую модель данных.

**Результат:** реализованы:

- схема БД;
- миграции;
- ограничения целостности;
- seed-данные;
- начальный каталог.

**Definition of Done:** база данных разворачивается автоматически и полностью готова к работе.

## Этап 2. Publication Service

**Цель:** реализовать публикацию рабочих каталогов продавцов.

Publication Service обеспечивает:

- импорт рабочего каталога;
- валидацию;
- сравнение изменений;
- обновление SellerProduct;
- обновление фотографий;
- создание CatalogPublication;
- обновление текущего состояния Seller.

Все изменения выполняются одной транзакцией.

**Definition of Done:** продавец может самостоятельно опубликовать каталог.

## Этап 3. REST API

**Цель:** предоставить единый API для пользовательских интерфейсов.

API реализует:

- просмотр каталога;
- просмотр товаров;
- просмотр категорий;
- публикацию каталога;
- получение истории публикаций.

**Definition of Done:** все пользовательские интерфейсы работают исключительно через REST API.

## Этап 4. Buyer Web

**Цель:** создать минимальный пользовательский интерфейс покупателя.

Покупатель может:

- просматривать категории;
- искать товары;
- просматривать предложения продавцов;
- просматривать карточки товаров.

**Definition of Done:** покупатель способен найти необходимый товар и выбрать продавца.

## Этап 5. Seller Cabinet

**Цель:** создать кабинет продавца.

Продавец может:

- скачать рабочий каталог;
- загрузить новую редакцию;
- просмотреть ошибки публикации;
- просмотреть историю публикаций.

**Definition of Done:** все операции публикации выполняются продавцом самостоятельно.

## Этап 6. Admin Cabinet

**Цель:** создать административный интерфейс.

Администратор может:

- управлять ProductGroup;
- управлять Product;
- выполнять модерацию;
- активировать и деактивировать продавцов.

**Definition of Done:** все административные операции выполняются через интерфейс.

## Контрольные точки

- **M1** — база данных успешно развёрнута.
- **M2** — первый каталог успешно опубликован.
- **M3** — первый каталог отображается покупателю.
- **M4** — первый продавец самостоятельно обновил каталог.
- **M5** — первый покупатель использует каталог.
- **M6** — получена первая регулярная выручка. Stage 1 считается успешно завершённым.

## Правила реализации

До завершения Stage 1 запрещается добавлять функциональность второго этапа — полный перечень см. [Scope.md](Scope.md). Любое исключение оформляется отдельным Architecture Decision Record (ADR).

## Критерии успешного завершения Stage 1

Stage 1 считается завершённым, если одновременно выполняются следующие условия:

- база данных функционирует в промышленной конфигурации;
- продавцы самостоятельно публикуют каталоги;
- покупатели регулярно используют каталог;
- администраторы сопровождают каталог без участия разработчиков;
- проект получает регулярную коммерческую выручку.

После достижения данных критериев проект переходит к проектированию Stage 2.

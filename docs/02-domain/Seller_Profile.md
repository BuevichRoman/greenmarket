# GreenMarket Seller Profile

**Version:** 1.0
**Status:** Normative

Настоящий документ закрывает OQ-003 (открытый вопрос PRD-000 v0.2).

## 1. Назначение

Seller Profile — доменная модель, описывающая информацию о продавце в GreenMarket.

Документ разделяет два уровня:

1. Доменная модель — что вообще может существовать в Seller Profile как сущности, независимо от текущей стадии продукта.
2. Требования Stage 1 — какое подмножество полей обязательно для MVP.

Смешение этих уровней преждевременно фиксирует решения MVP как архитектурные ограничения домена, поэтому они разделены явно.

## 2. Архитектурный статус модели

Это ключевое архитектурное решение PR-009, определяющее все остальные разделы документа.

- Seller Profile является самостоятельной доменной сущностью GreenMarket.
- Профиль существует независимо от публикаций каталога.
- Каталог содержит только товары.
- Seller Profile содержит информацию о продавце.
- Между Seller и Seller Profile существует связь один-к-одному.
- Seller является технической сущностью платформы (учётной записью продавца), а Seller Profile — публичной доменной моделью, содержащей информацию, используемую покупателями.
- Изменение Seller Profile не требует новой публикации каталога.
- Публикация нового каталога не изменяет Seller Profile.

### Архитектурное размещение

Seller Profile является самостоятельной доменной сущностью, логически отделённой от Catalog и Publication, и логически привязанной к Seller через `sellerId` — а не встроенным (embedded) набором полей внутри записи Seller и не частью Publication/Catalog.

Причины:

- профиль изменяется независимо от каталога и от публикаций;
- профиль должен быть доступен даже при отсутствии активных публикаций;
- один источник данных (например, регистрация) не должен быть единственным местом хранения — профиль переживает источник, из которого он был изначально создан.

Вопрос о конкретной реализации (отдельная таблица, отдельный сервис, отдельный Aggregate Root в терминах DDD и т.п.) остаётся открытым и оставлен на усмотрение Architecture Overview; на уровне домена фиксируется только логическое отделение от Catalog/Publication, а не конкретное решение реализации.

## 3. Market как отдельная сущность

`marketName` и `marketAddress` не являются полями продавца. Они относятся к сущности Market, чтобы избежать дублирования одного и того же адреса у сотен продавцов одного рынка.

```text
Market
  1
  │
  │
  ∞
Seller Profile
```

Один Market может содержать множество Seller Profile.

### Market

| Поле | Назначение | Тип | Обязательность |
|---|---|---|---|
| `marketId` | Идентификатор рынка | UUID/string | Обязательно |
| `name` | Название рынка | string | Обязательно |
| `address` | Адрес рынка | string | Обязательно |

Seller Profile ссылается на рынок через `marketId`, а не хранит `marketName`/`marketAddress` напрямую. `row` и `place` остаются в Seller Profile, так как это координаты конкретного продавца внутри рынка, а не свойства самого рынка.

> Примечание: модель Market не входит в объём данного PR как отдельный документ — здесь фиксируется только сам факт разделения сущностей и внешний ключ `marketId` в Seller Profile. Полная модель Market может быть описана отдельным PR при необходимости.

## 4. Доменная модель Seller Profile

Это полный набор полей, которые может содержать Seller Profile как сущность, вне зависимости от стадии продукта. На уровне домена большинство полей — опциональны; обязательность для конкретной стадии определяется отдельно в разделе 5.

| Поле | Назначение | Тип данных | Обязательность на уровне домена |
|---|---|---|---|
| `sellerId` | Идентификатор продавца | UUID/string | Обязательно |
| `name` | Название продавца | string | Обязательно |
| `status` | Статус профиля | enum (расширяемый) | Обязательно |
| `shortDescription` | Краткое описание | string | Опционально |
| `photo` | Фотография | string (URL), nullable | Опционально |
| `logo` | Логотип | string (URL), nullable | Опционально |
| `phone` | Телефон | string (E.164) | Опционально |
| `whatsapp` | WhatsApp | string (E.164) | Опционально |
| `marketId` | Ссылка на Market | UUID/string, nullable | Опционально |
| `row` | Ряд на рынке | string, nullable | Опционально |
| `place` | Место на рынке | string, nullable | Опционально |
| `workingHours` | Часы работы | string (например, `09:00–18:00`) | Опционально |
| `socialLinks` | Ссылки на соцсети | array<string (URL)> | Опционально |
| `email` | E-mail | string | Опционально |
| `additionalContacts` | Доп. способы связи | string | Опционально |
| `registeredAt` | Дата регистрации | datetime | Обязательно (системное) |
| `updatedAt` | Дата обновления профиля | datetime | Обязательно (системное) |

Опциональность на уровне домена сделана намеренно: доменная модель должна допускать, например, продавца без физической точки на рынке или без WhatsApp — такие случаи не должны требовать изменения модели данных в будущем, даже если Stage 1 их не использует.

> Примечание по `workingHours`: в будущих версиях допускается замена строкового представления на структурированную модель расписания без изменения остальных разделов документа.

### Поля вне модели Seller Profile

- **`specialization`** — не включается в модель. Специализация продавца выводится из состава его каталога (какие товары/категории он публикует) и не должна храниться дублирующим образом в профиле. Если в будущем потребуется явная специализация, независимая от каталога (например, для продавцов без публикаций), это отдельное решение вне PR-009.

## 6. Источник данных

Для каждого поля определяется **первичный источник** — то есть источник, из которого поле обычно поступает в момент создания профиля. Это не означает, что поле не может быть впоследствии скорректировано другим путём (например, администратором).

| Поле | Первичный источник | Возможна последующая корректировка |
|---|---|---|
| `sellerId`, `registeredAt`, `updatedAt` | Системное поле | Нет (генерируются платформой) |
| `name`, `shortDescription`, `phone`, `whatsapp`, `photo`, `logo`, `workingHours`, `socialLinks`, `email`, `additionalContacts` | Регистрация продавца | Да — администратором |
| `marketId`, `row`, `place` | Ввод администратором | Да — администратором |
| `status` | Системное поле (по умолчанию) | Да — администратором |
| `Market.name`, `Market.address` | Ввод администратором | Да — администратором |

Правило: для каждого поля должен быть однозначно определён именно первичный источник; допускается, что фактическое значение впоследствии меняется другим участником процесса (чаще всего — администратором).

## 7. Правила отображения

Каждое поле относится ровно к одной из категорий:

- **Отображается покупателю** — видно в Customer UI.
- **Отображается только администратору** — видно только в административном интерфейсе.
- **Используется только системой** — не отображается напрямую (`sellerId`, `registeredAt`, `updatedAt`).

Все поля из разделов 4–5, кроме `sellerId`, `status`, `registeredAt`, `updatedAt`, отображаются покупателю (при наличии значения). `status` влияет на отображение косвенно: профили не в статусе `ACTIVE` не показываются покупателю (см. раздел 9).

## 8. Статус профиля (`status`)

Поле называется `status`, а не `activityStatus`, так как в дальнейшем набор значений будет расширяться за пределы "активен/не активен".

Значения на Stage 1:

- `ACTIVE`
- `INACTIVE`

Зарезервировано на будущее (не реализуется в этом PR, приведено для совместимости именования):

- `BLOCKED`
- `UNDER_MODERATION`
- `SUSPENDED`

## 9. Customer UI — минимальный состав карточки продавца (Stage 1)

- название (`name`);
- фотография (`photo`), если имеется; при её отсутствии допускается использование логотипа (`logo`);
- краткое описание (`shortDescription`), если есть;
- контакты (`phone`, `whatsapp`, опционально `email`, `socialLinks`, `additionalContacts`);
- расположение (`Market.name`, `Market.address` по `marketId`, `row`, `place`);
- часы работы (`workingHours`).

Профили со `status ≠ ACTIVE` не отображаются покупателю.

## 10. Поля, допустимые для использования в поиске и фильтрации

Seller Profile не определяет поисковый механизм (полнотекстовый поиск, семантический поиск, конкретный движок и т.п.) — это предмет отдельного продуктового/технического документа. Здесь фиксируется только то, какие поля домена допускается использовать для поиска, фильтрации и сортировки, если/когда соответствующий механизм будет реализован.

- Допускается использовать в поиске: `name`, `shortDescription`.
- Допускается использовать в фильтрации: `Market.name`, `Market.address` (через `marketId`), `workingHours`, `status` (для покупателя — всегда фильтр по `ACTIVE`).
- Допускается использовать в сортировке: `name`, `Market.name`, `registeredAt` (административная сортировка).

## 11. Не входит в PR

- рейтинги;
- отзывы;
- чат;
- сообщения;
- звонки через GreenMarket;
- онлайн-статус;
- статистика продавца;
- личный кабинет продавца;
- редактирование профиля;
- отдельная нормативная модель Market (фиксируется только факт разделения и `marketId`).

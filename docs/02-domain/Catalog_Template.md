# GreenMarket Catalog Template

**Version:** 1.0
**Status:** Normative

## Назначение

Настоящий документ определяет формат шаблона рабочего каталога продавца — единственного инструмента ведения ассортимента на первом этапе GreenMarket. Документ дополняет [Publication_Model.md](Publication_Model.md) и [Publication_Service.md](../04-services/Publication_Service.md) деталями клиентского формата, на которые эти документы ссылаются.

В Stage 1 шаблон реализован как статическая таблица Google Sheets, предоставляемая GreenMarket. Продавец создаёт собственную копию шаблона, заполняет её и публикует, указав ссылку на таблицу — GreenMarket читает её через Service Account и никогда не пишет в неё (см. CR-001, `docs/06-development/adr/0002-static-google-sheets-template.md`). Архитектура описывает структуру шаблона (листы, поля, служебные данные), а не конкретный формат файла: смена формата (CSV, Google Sheets API и т.п.) в будущих версиях не потребует пересмотра остальных нормативных документов, ссылающихся на этот. Рабочий каталог — офлайн-представление, предназначенное только для редактирования бизнес-данных; сервер остаётся единственным источником истины по всем служебным данным.

## Нормативный артефакт

Начиная с PR-008 нормативным артефактом шаблона является файл
[`catalog_template_v1.xlsx`](templates/catalog_template_v1.xlsx), хранящийся
в этом репозитории. Файл версионируется в Git, открывается и в Excel, и в
Google Sheets без потери структуры, используется в автоматических тестах
Parser/Validator/Mapper (`backend/tests/test_catalog_template_artifact.py`) и
не зависит от конкретного Google-аккаунта или прав доступа.

Официальная Google Sheets Template, которую продавцы копируют себе через
«Создать копию», — производный артефакт: импортируется из этого файла в
Google Drive и может быть пересоздана из него в любой момент. Google-таблица
источником истины не является.

Примеры заполненного шаблона (для разработчиков и колле-ревью, не для
продавцов): [`templates/examples/catalog_template_v1_partial.xlsx`](templates/examples/catalog_template_v1_partial.xlsx),
[`templates/examples/catalog_template_v1_full.xlsx`](templates/examples/catalog_template_v1_full.xlsx).

Файл порождается детерминированно скриптом `backend/app/catalog_template/build.py`
(`uv run python -m app.catalog_template.build` из `backend/`) — при изменении
контракта колонок в `structure_validator.py` шаблон нужно перегенерировать и
закоммитить заново, а не редактировать вручную.

## Архитектурные принципы

Рабочий каталог (шаблон) создаётся продавцом как копия статического шаблона GreenMarket — сервер документ не генерирует и не обновляет его после публикации (CR-001). Обратная синхронизация в Stage 1 не реализуется: Google Sheets — источник публикации, не зеркало состояния GreenMarket. Файл содержит пользовательские данные и минимум служебной информации, необходимой для синхронизации с сервером; остальные служебные данные хранятся только на сервере.

## Структура рабочего каталога

Рабочий каталог состоит из нескольких листов.

### Лист «Каталог»

Основной лист. Одна строка = одна позиция продавца.

**Пользовательские поля:** наименование продавца, товарная группа GreenMarket, товарная позиция GreenMarket, цена, единица продажи, остаток, описание, дополнительные характеристики.

**Служебное поле (скрытое):** `SellerProductId`.

### Лист «Товарные группы»

Актуальный справочник ProductGroup — используется для выпадающих списков, продавцом не редактируется.

### Лист «Товарные позиции»

Актуальный справочник Product — используется для выпадающих списков, продавцом не редактируется.

### Лист «Инструкция»

Краткие правила: добавление/удаление/изменение товара, выбор группы/позиции, публикация.

### Лист `_System`

Метаданные шаблона (не конкретной публикации, CR-001): `TemplateVersion`, `TemplateId`. Продавцом не редактируется. `PublicationKey`/`CatalogHash` в документе не хранятся — оба генерируются сервером при публикации (см. `docs/04-services/Publication_Service.md`).

## SellerProductId

Постоянный технический идентификатор строки рабочего каталога — совпадает с `id` соответствующей записи SellerProduct (см. [Physical_Model.md](../03-database/Physical_Model.md)). Создаётся сервером, никогда не изменяется, принадлежит одной позиции продавца, используется только механизмом публикации, скрыт от пользователя.

## PublicationKey

`PublicationKey` — внутренний идентификатор конкретной публикации GreenMarket (CR-001). Генерируется сервером при каждом успешном вызове `POST /api/v1/publications`, хранится в `CatalogPublication.publication_key` и `Seller.current_publication_key`. Документ Google Sheets не содержит `PublicationKey` и не участвует в его генерации или проверке.

## Добавление, изменение и удаление товаров

**Добавление.** Новая строка без `SellerProductId` → при публикации сервер создаёт позицию и присваивает идентификатор.

**Изменение.** Любые пользовательские поля можно менять; сопоставление строк — по `SellerProductId`, не по наименованию.

**Изменение товарной позиции GreenMarket** для существующего товара трактуется как новая заявка на классификацию: `moderation_status` сбрасывается в `WAIT_PRODUCT`, поля `moderator_id`/`moderated_at`/`moderation_comment` очищаются, требуется повторная модерация.

**Удаление.** Удаление строки из рабочего каталога равносильно удалению позиции из опубликованного каталога (`is_published = false`); история публикаций сохраняется.

## Проверка публикации

Перед публикацией сервер проверяет: актуальность версии и `PublicationKey`; корректность структуры документа и обязательных полей; существование выбранных групп/позиций; корректность и непротиворечивость служебных идентификаторов (`SellerProductId`, отсутствие дублирования); целостность документа (`CatalogHash`).

Публикация — атомарная операция: при любой ошибке публикация полностью отклоняется, предыдущая версия каталога остаётся без изменений. Сервер возвращает полный перечень ошибок (строка + описание причины) в ответе `HTTP 422` — см. [Publication_Workflow.md](../04-services/Publication_Workflow.md), этап Business Validation.

## Защита служебных данных

**На уровне файла:** служебные поля и лист `_System` скрыты и защищены средствами Excel/Sheets от редактирования.

**На уровне сервера:** сервер никогда не доверяет служебным данным рабочего каталога без собственной проверки — проверяется версия документа, принадлежность `SellerProductId` продавцу, отсутствие дублирования, целостность документа, контрольная сумма `CatalogHash`.

## Обновление рабочего каталога

После каждой успешной публикации сервер формирует новую редакцию файла для продавца: актуальный ассортимент, изменения продавца, результаты обработки товаров, ожидавших модерации, новые доступные группы/позиции, новые `SellerProductId` (если были присвоены), новую служебную информацию (`_System`, новый `PublicationKey`).

## Правила работы продавца

Продавец работает только с последней полученной редакцией; редактирует только пользовательские данные; публикует только каталог, сформированный сервером (не произвольный файл); при отклонении публикации — исправляет ошибки по отчёту и повторяет попытку с той же редакцией (без смены `PublicationKey`, если ошибка не связана с версией).

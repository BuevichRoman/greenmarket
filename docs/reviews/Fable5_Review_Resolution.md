## Review Resolution

### R1. Platform Integration

**Замечание**

Недостаточно описана существующая платформа (Seller/User/Photo).

**Решение**

Accepted. Следует добавить документ `Platform_Integration.md`, в котором описать:

- что уже существует;
- какие таблицы используются;
- какие сервисы используются;
- что находится вне GreenMarket.

---

### R2. Revenue Model

**Замечание**

Definition of Done требует регулярной выручки. Документация не объясняет её источник.

**Решение**

Accepted. Следует добавить раздел `Revenue Hypothesis` в `Vision.md`. Необходимо определить:

- кто платит;
- за что платит;
- почему платит;
- как измеряется регулярность.

---

### R3. Publication Entity

**Замечание**

FSM содержит состояния публикации. REST API возвращает состояние публикации. Но модель данных хранит только успешные публикации.

**Решение**

Accepted. Необходимо принять архитектурное решение. Возможные варианты:

1. новая сущность Publication;
2. изменение CatalogPublication;
3. временное хранилище процесса публикации.

До принятия решения документация считается неполной.

**Статус: Resolved.** Принят вариант, не входивший в исходный список: публикация переведена из асинхронного процесса с промежуточными статусами в синхронную атомарную операцию в рамках одного HTTP-запроса и одной транзакции — отдельное хранилище статусов публикации не требуется, `CatalogPublication` остаётся журналом успешных публикаций без изменений. Обновлены документы `Publication_FSM.md` (переименован в `Publication_Workflow.md` и переписан как workflow), `Publication_Service.md`, `REST_API.md`, `Catalog_Template.md`, `Seller_MVP.md`, `Publication_Model.md`.

### R4. Bootstrap

**Замечание**

Bootstrap выглядит привязанным к PHP.

**Решение**

Accepted. Bootstrap должен определять только требования к окружению. Backend должен оставаться технологически нейтральным.

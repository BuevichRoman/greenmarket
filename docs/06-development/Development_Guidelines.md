# GreenMarket Development Guidelines

**Version:** 1.0
**Status:** Normative

## Назначение

Настоящий документ определяет обязательные правила разработки GreenMarket Stage 1. Все изменения системы должны соответствовать данным правилам. Цель документа — обеспечить единообразие реализации, предсказуемость архитектуры и простоту сопровождения проекта.

Этот документ заменяет собой отдельный ADR как основной контракт между архитектурой и разработкой; журнал архитектурных отклонений от Stage 1 ведётся в [adr/](adr/README.md).

## Основные принципы

Полный список с определениями — [Architecture_Principles.md](../00-overview/Architecture_Principles.md). Ключевые для разработки: Database First, API First, One Source of Truth, One Owner per Data, Keep MVP Small, Simplicity Before Flexibility.

## Источник истины

Для каждой категории данных существует единственный источник истины.

| Данные | Источник |
|---|---|
| Product | Product |
| Категории | ProductGroup |
| Коммерческие данные | SellerProduct |
| История публикаций | CatalogPublication |
| Текущее состояние каталога | Seller |
| Рабочий каталог | Excel продавца |

Дублирование ответственности запрещается.

## Владение данными

| Таблица | Владелец |
|---|---|
| ProductGroup | Admin |
| Product | Admin |
| SellerProduct | Publication Service |
| SellerProductPhoto | Publication Service |
| CatalogPublication | Publication Service |
| Seller | Platform |

Другие компоненты могут читать данные, но не изменять их.

## Правила разработки

Новая функциональность должна использовать существующую модель данных, существующий REST API, существующие бизнес-сущности. Создание новых сущностей допускается только после архитектурного решения ([ADR](adr/README.md)).

## Работа с базой данных

Изменение схемы допускается только через миграции. Редактирование существующих миграций запрещено. SQL должен соответствовать [DDL_Specification.md](../03-database/DDL_Specification.md).

## Работа с API

Все пользовательские интерфейсы работают только через REST API. Прямой доступ UI к базе данных запрещён.

## Работа с UI

Интерфейс не содержит бизнес-логики. Все решения принимаются серверной частью. UI отвечает только за отображение данных и взаимодействие с пользователем.

## Работа с Publication Service

Publication Service является единственным компонентом, изменяющим опубликованный каталог продавца. Любые обходные механизмы запрещены.

## Логирование

Все ошибки публикации должны журналироваться. Все критические операции должны иметь возможность последующего аудита.

## Производительность

Оптимизация выполняется только после измерений. Добавление кэшей, индексов и специальных структур данных без подтверждённой необходимости запрещается.

## Расширение проекта

До завершения Stage 1 запрещается реализовывать функциональность второго этапа — полный перечень см. [Scope.md](../01-overview/Scope.md). Все предложения по расширению помещаются в Backlog Stage 2.

## Code Review

Каждое изменение должно отвечать на три вопроса: соответствует ли архитектуре? не нарушает ли границы ответственности? можно ли реализовать проще? При отрицательном ответе хотя бы на один вопрос изменение требует доработки.

## Завершение Stage 1

Разработка первого этапа считается завершённой, если: документация соответствует реализации; миграции проходят автоматически; Publication Service работает; Buyer Web отображает опубликованный каталог; продавцы самостоятельно публикуют каталоги; получена первая регулярная выручка.

## Изменение настоящего документа

Настоящий документ изменяется только одновременно с изменением архитектуры проекта. Любые локальные соглашения команды не могут противоречить настоящему документу.

# 0001. Резервирование архитектурного направления: модель партии товара (Product Batch)

**Дата:** 2026-07-05
**Статус:** Принято

## Контекст

При обсуждении дальнейшего развития GreenMarket поднят вопрос о будущем расширении `SellerProduct` (предложения продавца) до модели партии товара — с атрибутами конкретной партии (дата сбора, срок хранения, органолептические характеристики, остаток и отзывы именно по данной партии). Такая функциональность не нужна для проверки основной бизнес-гипотезы Stage 1, но при проектировании физической модели данных и API стоит не принимать решений, которые заблокируют последующее расширение.

## Решение

В Stage 1 модель партии товара не реализуется: без новой таблицы, без изменения DDL, без изменения REST API, без изменения UI. Зафиксировано только архитектурное намерение:

- добавлен [`docs/02-domain/Future_Evolution.md`](../../02-domain/Future_Evolution.md) — описывает направление развития `SellerProduct` в сторону модели партии товара как материал для Stage 2;
- в [`Scope.md`](../../01-overview/Scope.md) добавлена новая подсекция "Партии товара" в разделе "Что не входит в Stage 1".

## Последствия

- `Scope.md` дополнен (нормативный документ, изменение требует ADR — оформлено данным ADR).
- Разработчики Stage 1 знают о планируемом направлении и не принимают решений (например, в DDL или REST API), которые заблокируют модель партии товара в Stage 2.
- Реализация партии товара как таковой остаётся полностью вне Stage 1 и не влияет на текущие миграции/API.
